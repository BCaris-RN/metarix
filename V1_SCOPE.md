# MetaRix V1 Scope

## 1. V1 objective

MetaRix V1 exists to prove a bounded, enterprise-credible operator workflow for governed social media operations.

The goal of V1 is not channel breadth.

The goal of V1 is workflow integrity.

## 2. V1 release statement

V1 will prove that MetaRix can support:

- strategy capture
- campaign and editorial planning
- content organization
- post draft workflow
- approval workflow
- bounded scheduling posture
- baseline reporting
- operator evidence visibility

## 3. V1 product slices

### Slice A — Strategy workspace

Purpose:
Capture the operating context that drives downstream social execution.

Required capabilities:
- business goals
- social goals
- target personas
- competitor catalog
- SWOT structure
- social audit structure
- benchmark baseline
- content pillars
- channel selection posture

V1 output:
A structured strategy record that can be referenced by campaigns and reports.

### Slice B — Campaign and planning workspace

Purpose:
Turn strategy into planned work.

Required capabilities:
- campaign creation
- campaign objective
- date windows
- channel mapping
- content pillar assignment
- editorial calendar structure
- weekly schedule structure
- evergreen content inventory
- asset references
- draft post creation

V1 output:
A usable calendar and content workspace tied to real campaign objects.

### Slice C — Draft and approval workflow

Purpose:
Move content through a bounded control path before scheduling or execution.

Required capabilities:
- draft status model
- reviewer assignment
- approval decision
- revision request
- approval history
- rejection reason
- scheduled-ready posture

V1 output:
A clear content state machine from draft to approved.

### Slice D — Scheduling posture

Purpose:
Prove bounded scheduling workflow without requiring full omnichannel complexity on day one.

Required capabilities:
- scheduled timestamp
- target channel assignment
- pre-schedule validation
- publish-eligibility posture
- schedule conflict awareness
- cancellation / reschedule support
- dry-run posture

V1 output:
A controlled schedule record ready for future execution depth.

### Slice E — Baseline reporting

Purpose:
Make operators capable of measuring performance and documenting learnings.

Required capabilities:
- reporting period
- channel-level metrics
- benchmark comparison
- standout results
- structured takeaways
- overall learnings
- future strategies
- continue / stop / start actions

V1 output:
A manager-readable report built from structured inputs.

### Slice F — Evidence visibility

Purpose:
Make major operator actions visible, reviewable, and bounded.

Required capabilities:
- approval evidence
- status history
- scheduling evidence
- denial visibility where applicable
- report generation trace
- current publish posture label

V1 output:
A first-pass operator evidence surface.

## 4. V1 explicit exclusions

The following are excluded from V1 unless directly promoted by a later scope decision:

- broad multi-network live publishing depth
- advanced reply inbox and full social care handling
- broad direct-message orchestration
- high-volume ad campaign management
- fully autonomous AI content generation and posting
- deep influencer relationship management
- advanced crisis automation
- warehouse-scale attribution modeling
- cross-channel paid/organic unified attribution
- production-grade recommendation automation
- Caris doctrine authoring inside Phoenix

## 5. V1 delivery rules

### Rule 1
One bounded end-to-end operator path is more valuable than many partial surfaces.

### Rule 2
Every major V1 capability must connect to a structured domain object.

### Rule 3
Every major V1 workflow state must be named, visible, and testable.

### Rule 4
Every bounded execution surface must be compatible with future Caris policy enforcement.

### Rule 5
The UI must not imply publish authority where none exists.

## 6. Recommended V1 folder posture

Suggested top-level product structure for the MetaRix repo:

- `docs/`
- `caris/`
- `phoenix/`
- optional local `8gentiC/` projection only where project-specific orchestration manifests are needed

## 7. V1 acceptance criteria

V1 is acceptable when all of the following are true:

1. A user can create a strategy record.
2. A user can define at least one audience persona and one competitor set.
3. A user can create a campaign tied to channel and date range.
4. A user can create and edit draft posts tied to a campaign and content pillar.
5. A user can move a draft through review and approval states.
6. A user can assign a schedule target and see bounded schedule posture.
7. A user can produce a structured report for a period.
8. A user can review baseline evidence for approval and scheduling state.
9. The implementation preserves 8gentiC / Caris / Phoenix boundaries.

## 8. V1 backlog order

### Priority 1
- strategy record
- campaign model
- content pillar model
- draft post model
- status workflow

### Priority 2
- approval workflow
- editorial calendar
- schedule posture
- asset linkage

### Priority 3
- baseline report model
- benchmark comparison
- learnings and next-step structure

### Priority 4
- evidence visibility
- denial surfaces
- release-ready expansion hooks

## 9. V1 release narrative

MetaRix V1 should be presented as:

“A governed planning, approval, scheduling, and reporting foundation for enterprise social operations.”

It should not be presented as:

- a complete enterprise social suite
- a full competitor replacement on day one
- an autonomous posting engine
- a broad AI marketing platform

## 10. Exit criteria for moving beyond V1

MetaRix should not widen into V1.5 or V2 until the following are true:

- core domain objects are stable
- approval and scheduling states are validated
- reporting outputs are usable
- evidence posture is legible
- repo boundaries remain clean
- the bounded operator path is demonstrable end to end
